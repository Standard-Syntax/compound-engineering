## Overview

Implement two remaining phases from the context engineering gap analysis (contextSpec.md):
- **Phase 5**: A new `/ce:guided` slash command that orchestrates the full CE pipeline with mandatory human approval gates between each phase
- **Phase 7**: Sub-agent specialization — split the generic `repo-research-analyst` into three focused agents (`codebase-locator`, `codebase-analyzer`, `codebase-pattern-finder`)

Phase 5 builds on Phase 1–4 (already completed). Phase 7 refines the research agents used by Phase 1.

## Problem Statement

### Phase 5: No Guided Human-gated Workflow

`/lfg` and `/slfg` run autonomously. For complex or brownfield work, autonomous execution skips the highest-leverage review points: research before planning, plan before implementation. The HumanLayer leverage hierarchy: bad research → thousands of bad code lines; bad plan → hundreds; bad code → one bad line.

CE has no command that forces human review at these decision points.

### Phase 7: Research Agents Are Too General-purpose

`repo-research-analyst` tries to do too much in one pass — locate files, understand architecture, find patterns, assess conventions. This floods its context window and produces mixed-quality output. HumanLayer's pattern uses three focused agents: one finds WHERE things are, one explains HOW things work, one finds EXAMPLES of patterns.

## What We're NOT Doing

- **`/lfg` stays autonomous** — `/lfg` remains the fast-path option. `/ce:guided` supplements it, not replaces it.
- **No removal of `repo-research-analyst`** — deprecate it without removal. Existing callers continue working.
- **No runtime context utilization monitoring** — Gap 2 from contextSpec (token-counting context awareness) is deferred.
- **No `ce:guided` beta variant** — guided mode is additive; no staged rollout needed.

## Technical Approach

### Phase 5: `/ce:guided`

**New file:** `plugins/compound-engineering/skills/ce-guided/SKILL.md`

**Modified file:** `plugins/compound-engineering/README.md` (add command entry)

#### Session Manifest Pattern

Each guided session maintains a manifest file so independent commands pass state:

```
.context/compound-engineering/guided-sessions/<uuid>/manifest.json
```

```json
{
  "session_id": "<uuid>",
  "started_at": "ISO8601",
  "research_artifact": null,
  "plan_path": null,
  "work_branch": null,
  "review_output": null,
  "solution_path": null,
  "status": "in_progress"
}
```

**Atomic writes:** Write to `.tmp` file first, then rename to target path. This prevents state corruption if the write fails mid-operation.

**Session recovery:** If a session is abandoned mid-flow, the user can resume by invoking `/ce:guided --resume <uuid>` with the session ID. The skill reads the manifest, identifies the last completed phase, and offers to continue from there.

#### Gate Design Principles

Per human-in-the-loop best practices (LangGraph interrupt patterns, CrewAI Flows):

1. **Stop to wait** — each gate uses the platform's blocking question tool (`AskUserQuestion` in Claude Code). The orchestrator pauses and waits for user input before proceeding.
2. **Evidence first** — before each gate, the skill gathers evidence and presents a structured summary. The human decides with context, not blind.
3. **No contradictory rules** — gate choices are consistent across all phases. No phase says "skip confirmation" while another requires it.
4. **Non-contradictory resume** — if the orchestrator is interrupted and resumes, gate state is consistent with where it left off.

#### Gate Presentation Format

At each gate, present a structured summary (not the full artifact):

```
## Gate: [Phase Name] Complete

**Artifact:** `path/to/artifact.md`
**Status:** ✓ Complete / ⚠ Incomplete

### Evidence
- **Topic:** [from frontmatter]
- **Tags:** [from frontmatter]
- **Files referenced:** [count from research artifact]
- **Phases:** [count from plan artifact]

### Key Findings
[3–5 bullet points auto-generated from artifact body]

### What changed since last gate
[1-2 sentences on what was done in this phase]
```

**Blocking question:**
> "Ready at `path/to/artifact.md`. Review the evidence above. How would you like to proceed?"

Choices:
1. **Proceed** — move to the next phase
2. **Revise this phase** — re-run the current phase (overwrites the artifact)
3. **Exit session** — abandon the session

If the user chooses **Revise**, the orchestrator re-runs the same phase with the same arguments. If the phase fails repeatedly, the user can refine the arguments before re-running.

#### Command Invocation Formats

Each phase invokes an existing CE command. The invocation must match what the target skill actually accepts:

| Phase | Command | Invocation | Notes |
|-------|---------|------------|-------|
| 1 | `/ce:research` | `/ce:research $ARGUMENTS` | No changes needed |
| 2 | `/ce:plan` | `/ce:plan $ARGUMENTS` | Pass research artifact path as part of the argument string: `/ce:plan $ARGUMENTS\n\nResearch: docs/research/YYYY-MM-DD-slug.md`. `ce:plan` Step 0.5 already handles this. |
| 3 | `/ce:work` | `/ce:work docs/plans/YYYY-MM-DD-NNN-*-plan.md` | Pass plan path as argument. **No `--sub-mode` flag exists.** See Double-Gating Resolution below. |
| 4 | `/ce:review` | `/ce:review $BRANCH_NAME [--serial]` | `ce:review` is PR/branch-centric. Pass the branch name where work was committed. |
| 5 | `/ce:compound` | `/ce:compound` | No arguments needed |

**Important:** `/ce:plan` does not have a `--research` flag. The research artifact path is passed as part of the argument text, which `ce:plan` already parses in Step 0.5 (Research Artifact Check).

#### Double-Gating Resolution

`/ce:work` uses `human_interrupt_node` in the LangGraph engine, which calls `langgraph.types.interrupt()`. This is a core graph mechanism — there is no `--no-internal-gates` flag.

**Resolution:** Gate 3 (Post-Implementation Review) is the guided-mode gate. `/ce:work` runs with its normal internal gates intact. The human pauses twice per implementation phase (guided gate + work gate) — this is conservative but correct.

> **Tradeoff:** The plan originally proposed `--no-internal-gates` sub-mode. This does not exist and would require engine changes. The simpler approach: accept double-pause, which gives the human an extra checkpoint.

#### Orchestration Flow

```
/ce:guided <topic>
  │
  ├─ Generate session UUID
  ├─ Create manifest at .context/.../<uuid>/manifest.json
  │
  ├─ Run /ce:research <topic>
  │     └─ Write artifact path → manifest.research_artifact
  │
  ├─ GATE 1: Research Review
  │     ├─ Proceed → next
  │     ├─ Revise → re-run ce:research
  │     └─ Exit → end session
  │
  ├─ Run /ce:plan with research artifact in arguments
  │     └─ Write plan path → manifest.plan_path
  │
  ├─ GATE 2: Plan Review
  │     ├─ Proceed → next
  │     ├─ Revise → re-run ce:plan
  │     └─ Exit → end session
  │
  ├─ Run /ce:work <plan_path>
  │     └─ Write branch name → manifest.work_branch
  │     (Note: /ce:work has its own internal phase gates)
  │
  ├─ GATE 3: Post-Implementation Review
  │     ├─ Proceed → next
  │     ├─ Continue working → re-enter /ce:work
  │     └─ Exit → end session (artifacts persist)
  │
  ├─ Run /ce:review <branch_name>
  │     └─ Write review output → manifest.review_output
  │
  ├─ GATE 4: Review Findings
  │     ├─ Document → next
  │     ├─ Skip → end session
  │     └─ Exit → end session
  │
  └─ Run /ce:compound
        └─ Write solution path → manifest.solution_path
              └─ End session
```

```yaml
# /ce:guided — Guided Compound Engineering Workflow

## Session Initialization

1. Generate session UUID
2. Create manifest directory: `.context/compound-engineering/guided-sessions/<uuid>/`
3. Write manifest with `status: in_progress` (atomic write: write to `.tmp`, then rename)
4. Announce: "Starting guided session `[uuid]`. I'll pause at each phase for your review."

## Phase 1: Research

Run `/ce:research $ARGUMENTS`

After completion:
- Update manifest: `research_artifact` = path to produced artifact (atomic write)
- Proceed to Gate 1

## Gate 1: Research Review

Read the research artifact. Generate a structured summary:
- Extract `topic`, `tags`, `date`, `git_commit` from frontmatter
- Count `## Code References` entries → "N files referenced"
- Read `## Summary` section → use first 3 bullets

If the artifact is missing required sections (`topic`, `## Summary`, `## Code References`), note "⚠ Incomplete" in the status.

Present the structured gate format. Use `AskUserQuestion` (or platform equivalent) to block.

## Phase 2: Planning

Run `/ce:plan` with the research artifact path in the argument text:
```
/ce:plan <original topic>

Research artifact: docs/research/YYYY-MM-DD-slug.md
```

After completion:
- Update manifest: `plan_path` = path to produced plan (atomic write)
- Proceed to Gate 2

## Gate 2: Plan Review

Read the plan artifact. Generate a structured summary:
- Extract `title`, `type`, `date` from frontmatter
- Count `## Acceptance Criteria` checklist items
- Count `## Implementation Phases` sub-sections
- Read `## What We're NOT Doing` → list items

Present the structured gate format. Block with `AskUserQuestion`.

## Phase 3: Implementation

Run `/ce:work <plan_path>`

`/ce:work` executes the plan with its own internal `human_interrupt_node` phase gates. These fire independently of the guided-mode gates.

After completion:
- Update manifest: `work_branch` = branch name used for work (atomic write)
- Proceed to Gate 3

## Gate 3: Post-Implementation Review

Read the plan file. Count completed vs. pending checkboxes. Extract any `## Key Decisions` noted during work.

Present the structured gate format. Block with `AskUserQuestion`.

## Phase 4: Review

Run `/ce:review <branch_name>` where `branch_name` is derived from the work session.

After completion:
- Update manifest: `review_output` = summary of review findings (atomic write)
- Proceed to Gate 4

## Gate 4: Review Findings

Count total findings by severity. List top 3 high-severity findings.

Present the structured gate format. Block with `AskUserQuestion`.

## Phase 5: Compound

Run `/ce:compound`

After completion:
- Update manifest: `solution_path` = path to produced solution doc (atomic write)
- Set manifest `status: completed`
- Announce session summary

## Session End

Output:
```
## Guided Session Complete

**Session ID:** `<uuid>`
**Artifacts:**
- Research: `docs/research/...`
- Plan: `docs/plans/...`
- Solution: `docs/solutions/...`
```
```

#### Output Schemas (API Contract)

Each specialized agent returns structured JSON (not Markdown prose). The orchestrator validates the JSON against the schema before including it in the research artifact.

**`codebase-locator` output schema:**

```json
{
  "agent": "codebase-locator",
  "status": "complete",
  "search_strategy": "<what was searched>",
  "files": [
    { "path": "path/to/file.ext", "description": "1-line description" }
  ],
  "directories": [
    { "path": "path/to/module/", "contains": ["subdir1", "subdir2"] }
  ],
  "summary": {
    "total_files": 0,
    "total_directories": 0,
    "technologies": ["list from file extensions"]
  }
}
```

**`codebase-analyzer` output schema:**

```json
{
  "agent": "codebase-analyzer",
  "status": "complete",
  "components": [
    {
      "name": "<component_name>",
      "file": "path/to/file.ext:line_range",
      "responsibility": "1-sentence description",
      "dependencies": [
        { "name": "<dep>", "usage": "how used" }
      ],
      "public_interface": [
        { "signature": "function_name(args) -> return_type", "description": "what it does" }
      ],
      "side_effects": "none | reads: X | writes: Y | network: Z",
      "data_flow": ["Component A", "Component B", "Component C"]
    }
  ]
}
```

**`codebase-pattern-finder` output schema:**

```json
{
  "agent": "codebase-pattern-finder",
  "status": "complete",
  "pattern_name": "<what pattern was searched>",
  "instances": [
    {
      "file": "path/to/file.ext:line_range",
      "code": "<extracted code block>",
      "context": "<surrounding code explaining why this pattern was used>",
      "similarity": "high | medium | low"
    }
  ],
  "summary": {
    "total_instances": 0,
    "consistency": "highly consistent | some variation | inconsistent usage"
  }
}
```

**Script-first implementation:** Each agent includes a bundled script (e.g., `codebase-locator.py`) that produces the structured JSON output. The SKILL.md instructs the agent to run the script and present the JSON output — the model does not generate the JSON itself. This follows the script-first pattern: script does mechanical work, model presents.

#### Agent Implementations

**`codebase-locator`** (WHERE things are):
- Input: research question decomposed into 2–4 concrete search targets
- Tools: Glob, Grep, Read (on discovery passes only). Prefer dedicated file tools over shell.
- Constraint: Do NOT analyze, evaluate, or recommend. Report paths only.
- Termination: Returns when all targeted paths are found (or confirmed absent). Soft guideline: 3–5 tool-call rounds. If targets remain unfound after 5 rounds, return partial results with `status: partial`.
- Output: Structured JSON (see schema above)

**`codebase-analyzer`** (HOW things work):
- Input: specific file:line references from `codebase-locator` output
- Tools: Read (full files), optionally Grep for cross-references
- Constraint: Describe function signatures, data flows, dependencies. Do NOT suggest changes.
- Output: Structured JSON (see schema above)

**`codebase-pattern-finder`** (EXAMPLES of existing patterns):
- Input: pattern description (e.g., "how does the codebase handle retries?")
- Tools: Grep, Read
- Constraint: Find existing examples, report them verbatim. Do NOT evaluate whether the pattern is good.
- Output: Structured JSON (see schema above)

## Alternative Approaches Considered

### Guided Mode: Use `/lfg` with a flag (Rejected)

**Approach:** Add a `--guided` flag to `/lfg` that enables human gates.

**Why rejected:** `/lfg` is explicitly designed for autonomous chaining. Adding conditional gate logic inside the skill makes it harder to reason about. A separate `/ce:guided` skill is cleaner and easier to maintain.

### Sub-agents: Keep repo-research-analyst and add new ones alongside (Rejected)

**Approach:** Keep `repo-research-analyst` as-is, create new agents as additional optional agents.

**Why rejected:** This creates two parallel code paths. `ce:research` would have to choose. Deprecating the old one forces migration and keeps the codebase lean.

### Sub-agents: Replace repo-research-analyst atomically (Rejected)

**Approach:** Remove `repo-research-analyst`, update all callers.

**Why rejected:** Breaking change for external callers. Deprecation with backward compatibility is lower risk.

## System-Wide Impact

### Interaction Graph

- `/ce:guided` → calls `/ce:research` → produces artifact → manifest updated (atomic write)
- `/ce:guided` → calls `/ce:plan` with research path in argument text → produces plan → manifest updated
- `/ce:guided` → calls `/ce:work` with plan path → `/ce:work` runs with internal gates → manifest updated
- `/ce:guided` → calls `/ce:review` with branch name → produces findings → manifest updated
- `/ce:guided` → calls `/ce:compound` → produces solution → manifest updated
- `ce:research` → calls 6 agents (3 new + 3 existing) in parallel → produces artifact
- `codebase-locator/analyzer/pattern-finder` → used by `ce:research` initially; may be called directly by other skills later

### Error & Failure Propagation

- If `/ce:research` fails → Gate 1 pauses. User can revise arguments and re-run, or exit.
- If `/ce:plan` fails repeatedly → Gate 2 pauses. User can revise arguments or exit.
- If `/ce:work` fails → normal `/ce:work` error handling. User can resume.
- If any sub-agent times out → remaining 5 agents' outputs are included. Artifact notes which timed out.
- If manifest write fails → abort session. Do not proceed with unclear state.

### State Lifecycle Risks

- Session manifest persists in `.context/compound-engineering/guided-sessions/` — ephemeral scratch space
- Artifacts (research, plan, solution) persist in `docs/` — durable
- If user abandons mid-session → work artifacts persist in `docs/` and git branch. User can resume with `--resume <uuid>` or continue manually.
