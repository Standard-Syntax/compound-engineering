## Problem Statement

Plans generated by `/ce:plan` describe what to build and how to build it, but not what the system should look like after implementation is complete. Without an explicit end-state specification, the implementation agent makes incremental decisions about "done" at each phase. The end result may satisfy each phase's criteria while missing the overall goal.

ContextSpec Gap 10: "Without a clear end-state specification, the implementation agent makes incremental decisions about 'done' at each phase. The end-state section gives the agent a north star."

## What We're NOT Doing

- Adding "Desired End State" to brainstorm or ideation templates — this is a planning artifact concern only
- Retrofitting existing plan files — new plans generated after this change include the section; old plans are unchanged
- Changing the plan template order in a way that breaks existing tooling — section placement follows the existing section hierarchy

## Technical Approach

### File to Modify

`plugins/compound-engineering/skills/ce-plan/SKILL.md`

### Section Placement

In all three templates, add `## Desired End State` after `## Overview` (or equivalent — see per-template details below).

**Rationale:** `## Overview` describes the feature. `## Desired End State` describes the target. Placing end-state immediately after overview gives the agent the north star before it reads the problem statement or proposed solution.

### Per-Template Changes

#### MINIMAL Template

**Current structure:**
```
## Overview
[Brief description]

## Acceptance Criteria
```

**After change:**
```
## Overview
[Brief description]

## Desired End State

[The target state after implementation — what the system looks like, how to verify it's correct, and what success looks like in concrete terms]

## Acceptance Criteria
```

**Note:** MINIMAL has no `## Overview` heading per se — the title line is followed directly by `[Brief problem/feature description]`. Add `## Desired End State` before `## Acceptance Criteria`.

## Desired End State Content Guidance

The section should answer:

1. **What does the system look like after this is done?** (functional outcome)
2. **How do we know it's right?** (verification approach)
3. **What are the concrete success indicators?** (measurable criteria beyond acceptance criteria)

**Example structure:**

```markdown
## Desired End State

The system processes user requests in under 100ms p95 latency, with zero data loss on worker restart, and full observability via structured logs.

**Functional outcome:** [What the feature does after implementation]
**Verification approach:** [How to confirm correctness — test scenarios, smoke tests]
**Success indicators:**
- Latency: p95 < 100ms under [load profile]
- Correctness: [specific behavior verified]
- Observability: [logging/tracing requirements]
```

## Alternative Approaches Considered

### Place after Proposed Solution (Rejected)

**Approach:** Add `## Desired End State` after `## Proposed Solution`.

**Why rejected:** This puts end-state after the implementation plan. The agent has already committed to an approach by then. End-state should precede solution — it constrains the solution, not the other way around.

### Place at end of plan (Rejected)

**Approach:** Add as the last major section before acceptance criteria.

**Why rejected:** The north star should appear early, not buried at the end. Implementation agents may not read the full plan before starting.
