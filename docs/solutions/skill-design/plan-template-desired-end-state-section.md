# Plan Template Desired End State Section

## Origin

Gap 10 from the context engineering gap analysis (contextSpec.md). The ce:plan templates described what to build and how to build it, but not what the system should look like after implementation was complete. Without an explicit end-state specification, the implementation agent makes incremental decisions about "done" at each phase — the end result may satisfy each phase's criteria while missing the overall goal.

## Problem

**Symptom:** Plans generated by `/ce:plan` had no "Desired End State" section. Implementation agents received a roadmap of steps but no destination. This led to:
- Incremental "done" decisions at each phase that individually met criteria but collectively missed the overall goal
- No north star to evaluate whether implementation choices aligned with the intended outcome
- Difficulty determining when a feature was truly complete beyond passing acceptance criteria

**Root cause:** The plan template structure focused on inputs and process (what to build, how to build it, acceptance criteria) without defining the target state.

## Investigation

Studied the three ce:plan detail-level templates (MINIMAL, MORE, A LOT) in `plugins/compound-engineering/skills/ce-plan/SKILL.md`. Each had a different structure:

- **MINIMAL**: Title → brief description → Acceptance Criteria (no Overview heading)
- **MORE**: Overview → Problem Statement → Proposed Solution → ...
- **A LOT**: Overview → Problem Statement → Proposed Solution → Technical Approach → ...

None included an end-state specification. Gap 10 in contextSpec explicitly identified this as a LOW IMPACT, TRIVIAL effort gap worth closing.

## Solution

Added `## Desired End State` section to all three templates. The section answers:
1. What does the system look like after this is done?
2. How do we know it's right?
3. What are the concrete success indicators?

**MINIMAL template** (line 233): Section added before `## Acceptance Criteria`, with concise guidance:
```markdown
## Desired End State

[The target state after implementation — what the system looks like, how to verify it's correct, and what success looks like in concrete terms]

## Acceptance Criteria
```

**MORE template** (line 312): Section added after `## Overview`, with full guidance:
```markdown
## Overview

[Comprehensive description]

## Desired End State

[The target state after implementation. Include: what the system looks like, how to verify correctness, and what concrete success indicators exist. This is the north star for implementation decisions.]

## Problem Statement / Motivation
```

**A LOT template** (line 413): Section added after `## Overview`, with expanded guidance including functional outcomes and quality attributes:
```markdown
## Desired End State

[The target state after implementation. Describe: what the system looks like, how correctness is verified, and what concrete success indicators exist. This section is the north star — implementation decisions should be consistent with this state. Include both functional outcomes (what it does) and quality attributes (how well it performs, security posture, etc.).]

## Problem Statement
```

## Key Decision

**Placement: after Overview, before Problem Statement.**

Alternative approaches rejected:
- **After Proposed Solution** — agent has already committed to an approach by then; end-state should constrain the solution, not follow it
- **At end of plan** — north star should appear early; implementation agents may not read the full plan before starting

## Prevention Strategies

1. **Template sections as decision constraints.** New template sections placed early in the document (before Problem Statement, Proposed Solution) act as constraints on downstream decisions. Always consider section order when adding template structure.
2. **North star before roadmap.** When designing plan templates, describe the destination before the journey. Implementation agents can evaluate whether their choices align with the goal.
3. **Template gaps are compounding debt.** Small template omissions (missing sections) create larger problems during implementation. Review plan templates for completeness when closing gaps.
